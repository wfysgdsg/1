## 二十一、fetchAll 分页核心 Bug 修复（2026-05-25）

### 32. fetchAll 复用 Query 对象导致 skip/limit 失效

- **根因**：`fetchAll` 在循环中复用同一个 Query 对象，第一次 `.get()` 后后续 `.skip()` 和 `.limit()` 均失效，微信云数据库回退到默认 20 条限制，循环判定 `data.length < pageSize` 提前终止，只能获取 20 条
- **文件**：`utils/db.js`
- **修复**：
  1. 工厂函数模式 — `fetchAll` 参数支持 `function() { return db.collection(...).where(...) }`，每次循环调用工厂获取全新 query
  2. pageSize 上限 20 — `Math.min(options.pageSize || 20, 20)`，匹配微信单次返回的可靠上限
  3. 终止条件不变 — `data.length < pageSize` 才 break
- **影响范围（7 个文件，13 处调用全部改为工厂函数模式）**：
  - `pages/borrow/list.js`（2 处：借货明细 + 个人库存）
  - `pages/borrow/detail.js`（1 处：借货详情）
  - `pages/borrow/transfer-list.js`（1 处：调货历史）
  - `pages/sale/list.js`（1 处：销售汇总）
  - `pages/index/index.js`（2 处：今日销售 + 应收账款）
  - `pages/report/month.js`（2 处：本月 + 上月）
  - `pages/report/index.js`（1 处：报表）

## 三十六、借货部分开票 Bug 修复（2026-06-11）

### 106. 借货部分数量开票后整条记录消失

- **Bug**：单位借货 3 个商品 A，批量开票只卖 1 个，借货明细中该单位的商品 A 全部消失
- **根因 1**（前端）：`pages/sale/add.js` 的 `calcProfit()` 重建 `selectedGoods` 数组时丢掉了 `originalBorrowId` 字段，导致云函数收不到借货记录与销售数量的映射关系
- **根因 2**（云函数）：`cloudfunctions/submitSale/index.js` 导入模式处理借货记录时一刀切——不管卖多少，直接把 `importIds` 中所有借货记录标为 `status: 'sold'`
- **修复**：
  - `pages/sale/add.js` — `calcProfit()` 返回对象补上 `originalBorrowId: g.originalBorrowId`
  - `cloudfunctions/submitSale/index.js` — 从 `selectedGoods` 的 `originalBorrowId` + `quantity` 构建映射，对比售出数量与借货数量：售出 ≥ 借货 → 标 `sold`；售出 < 借货 → `_.inc(-soldQty)` 扣减，保持 `pending`，记录 `saleId` 供红冲追溯
  - `cloudfunctions/saleManage/index.js` — 红冲时查 `{ saleId }`（不限 status），对部分售出的记录恢复扣减的数量
